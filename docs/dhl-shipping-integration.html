INTERNAL TECHNICAL GUIDE

DHL Shipping Integration Guide

คู่มือ replicate การเชื่อมต่อ DHL ในระบบ e29-shop-api — สำหรับทีมที่ต้องการนำ pattern ไปใช้ในโปรเจกต์อื่น

Stack: Node.js · Express · Sequelize (MySQL) · axios · PM2 cron  |  เอกสารนี้สรุปจากโค้ดจริงในโปรเจกต์

⚠️ Security — อ่านก่อน: เอกสารนี้ระบุเฉพาะ ชื่อ ของ environment variable เท่านั้น ไม่มีค่า credential จริง ทีมที่นำไปใช้ต้องขอ DHL account / credential ของตัวเอง อย่า copy ค่าจาก .env ของโปรเจกต์นี้ และควรเก็บ secret ใน vault / CI secret ไม่ commit ลง git

1Overview — DHL สองระบบ

ระบบนี้เชื่อมต่อ DHL 2 ระบบที่แยกกันโดยสิ้นเชิง โดยเลือกอัตโนมัติจากประเทศปลายทางของออเดอร์ (ดูจาก country.cca2 ในตาราง sm_addr_countries):

ระบบใช้เมื่อDHL ProductAPI ที่เรียกAuthentication
DHL Local
eCommerce API		 ปลายทาง = ไทย
cca2 == 'TH'		 ในประเทศ (Domestic) OAuth Token (v1), Label (v2), Tracking (v3) Token-based
(clientId + password → accessToken)
DHL Global
Express / MyDHL API		 ปลายทาง = ต่างประเทศ
cca2 != 'TH'		 ระหว่างประเทศ (International, Parcel "P") POST /shipments (สร้าง label + invoice + customs ในครั้งเดียว) HTTP Basic Auth
(username + password)

หลักการเลือก: ตอน admin กดสร้าง label ระบบจะ resolve ประเทศปลายทางก่อน ถ้าเป็นไทยใช้ DHL Local ถ้าไม่ใช่ไทยใช้ DHL Global — logic นี้อยู่ใน controllers/v1/admin/order.js → createTracking() (บรรทัด ~100 และ ~176)

2Architecture & Data Flow

วงจรชีวิตของออเดอร์ตั้งแต่ลูกค้าสั่งจนสถานะอัปเดตอัตโนมัติ:

[1] ลูกค้า checkout │ worker: workers/manage.order.js ▼ [2] คำนวณค่าส่ง (shipping_fee) ├─ ไทย : เรต fixed ตามจำนวนชิ้น (80 / 100 / 150 บาท) └─ ต่างประเทศ : SmDHLService.calculateByRegion() → lookup ตาราง sm_rate_shippings ▼ [3] สร้าง PurchaseOrder (สถานะ prepare) ▼ [4] Admin กดสร้าง label (controllers/v1/admin/order.js → createTracking) ├─ TH → DHL Local : OAuth token → POST Label (v2) → ได้ PNG label (base64) └─ INTL→ DHL Global: POST /shipments → ได้ PDF label + invoice + tracking number ▼ [5] เก็บไฟล์ลง public/images/orders/{poid}/... + insert แถวลง purchase_order_ship_tracks ▼ [6] Cron 23:45 ทุกวัน (crons/update-dhl-completed.js) └─ ดึง tracking ของออเดอร์สถานะ prepare/shipped → map DHL event code → อัปเดตสถานะออเดอร์ event 77206 → shipped event 77093 → completed

4DHL Local (eCommerce) API

ใช้กับออเดอร์ปลายทางไทย ลำดับการเรียก: (a) ขอ token → (b) สร้าง label → (c) เช็ค tracking ทุก request ส่ง accessToken ใน header block (hdr) ของ body

(a) Get Access Token

GET {DHL_API_TOKEN}
  ?clientId={clientId}&password={password}&returnFormat=json

// response — ต้องเช็ค code == '100000'
accessTokenResponse.responseStatus.code   // '100000' = OK
accessTokenResponse.token                 // นำไปใส่ใน request ถัดไป

(b) Create Label

shipmentID สร้างจาก pattern THHNY{ลำดับ 2 หลัก}{poid} เช่น THHNY01PO12345 (ลำดับ = จำนวน tracking เดิม + 1)

POST {DHL_API_LABEL}
{
  "labelRequest": {
    "hdr": {
      "messageType": "LABEL",
      "messageDateTime": "<now>",
      "accessToken": "<token>",
      "messageVersion": "1.4",
      "messageLanguage": "th_TH"   // 'en' ถ้าไม่ใช่ไทย
    },
    "bd": {
      "pickupAccountId": "<pickupAccountId>",
      "soldToAccountId": "<soldToAccountId>",
      "handoverMethod": 1,
      "shipmentItems": [{
        "shipmentID": "THHNY01PO12345",
        "productCode": "ECO",
        "totalWeight": 1200,          // หน่วยกรัม
        "totalWeightUOM": "G",
        "insuranceValue": null,
        "currency": "THB",
        "consigneeAddress": {
          "name": "<ชื่อผู้รับ>",
          "address1": "...", "address2": "...", "address3": "<ตำบล>",
          "city": "<อำเภอ>", "state": "<จังหวัด>",
          "country": "TH", "postCode": "10110", "phone": "08xxxxxxxx"
        },
        "returnMode": "01"
      }],
      "label": { "format": "PNG", "layout": "1x1" }
    }
  }
}

// response — ต้องเช็ค code == '200'
labelResponse.bd.responseStatus.code        // '200' = OK
labelResponse.bd.labels[0].shipmentID       // = tracking_ref
labelResponse.bd.labels[0].content          // PNG เป็น base64 → เก็บเป็นไฟล์

(c) Tracking

ส่งได้ทีละหลายเลข (โค้ด cron batch ครั้งละ 100) ผ่าน array trackingReferenceNumber

POST {DHL_API_TRACKING}
{
  "trackItemRequest": {
    "hdr": { "messageType": "TRACKITEM", "accessToken": "<token>",
             "messageDateTime": "<now>", "messageLanguage": "th_TH" },
    "bd": {
      "pickupAccountId": "...", "soldToAccountId": "...",
      "ePODRequired": "Y",
      "trackingReferenceNumber": ["THHNY01PO12345", "THHNY01PO12346"]
    }
  }
}

// response — ต้องเช็ค code == '200'
trackItemResponse.bd.shipmentItems[].events[].status
//  '77206' = picked up / out for delivery → map เป็น "shipped"
//  '77093' = delivered                    → map เป็น "completed"

5DHL Global (Express / MyDHL) API

เรียกครั้งเดียวได้ครบ: สร้าง shipment + label + commercial invoice + customs declaration + ประเมินค่าส่ง ใช้ HTTP Basic Auth (axios auth: {username, password})

หมายเหตุ field hardcode ในโค้ด (ทีมอื่นต้องเปลี่ยนเป็นของบริษัทตัวเอง): ข้อมูล shipperDetails (ที่อยู่ผู้ส่ง, เบอร์, email, เลข VAT 0125563023507), account number 568963397, ขนาดกล่อง 35×38×1 cm, description: "Souvenir", incoterm: "DAP"

POST {DHL_GLOBAL_API_LABEL}        // Basic Auth: DHL_GLOBAL_USER / DHL_GLOBAL_PASSWORD
{
  "plannedShippingDateAndTime": "2026-06-26T10:00:00 GMT+07:00",
  "pickup": { "isRequested": false },
  "productCode": "P", "localProductCode": "P",
  "getRateEstimates": true,
  "estimatedDeliveryDate": { "isRequested": true },
  "accounts": [
    { "typeCode": "shipper", "number": "568963397" },
    { "typeCode": "payer",   "number": "568963397" }
  ],
  // WY = Paperless Trade waybill — เปิดเฉพาะประเทศที่ country.plt_allowed_in = true
  "valueAddedServices": [{ "serviceCode": "WY" }],
  "outputImageProperties": {
    "printerDPI": 300, "encodingFormat": "pdf",
    "imageOptions": [
      { "typeCode": "label",      "templateName": "ECOM26_84_001", "isRequested": true },
      { "typeCode": "waybillDoc", "templateName": "ARCH_8X4",      "isRequested": true },
      { "typeCode": "invoice",    "isRequested": true, "invoiceType": "commercial" }
    ]
  },
  "customerReferences": [{ "value": "<poid>", "typeCode": "CU" }],
  "customerDetails": {
    "shipperDetails":  { /* ที่อยู่ผู้ส่ง hardcode (บริษัท) + registrationNumbers VAT */ },
    "receiverDetails": { /* ที่อยู่ผู้รับจากออเดอร์ — postalCode ว่างถ้า country.no_postal_code */ }
  },
  "content": {
    "packages": [{ "weight": 1.2, "dimensions": { "length": 35, "width": 38, "height": 1 } }],
    "isCustomsDeclarable": true,
    "declaredValue": 50.00, "declaredValueCurrency": "USD",
    "exportDeclaration": {
      "lineItems": [ /* map จาก order_detail: description, price_usd, qty, weight(kg) */ ],
      "invoice": { "number": "<poid>", "date": "2026-06-20" },
      "placeOfIncoterm": "<จังหวัด>, <ประเทศ>"
    },
    "description": "Souvenir", "incoterm": "DAP", "unitOfMeasurement": "metric"
  }
}

Response fields ที่ใช้

dhlGenLabel.data.shipmentTrackingNumber          // = tracking_ref
dhlGenLabel.data.documents[]                      // typeCode: "label" | "invoice" | "waybillDoc"
   .content                                       // PDF base64 → เก็บเป็นไฟล์ .pdf
dhlGenLabel.data.shipmentCharges[]                // เลือกตัวที่ currencyType == 'BILLC'
   .price                                          // → เก็บลง shipment_charges

global ต่างจาก local ตรงที่: น้ำหนักส่งเป็น กิโลกรัม (totalWeight / 1000), ได้ไฟล์เป็น PDF, มี invoice แยก, มี shipment_charges

6Shipping Cost Calculation

คำนวณตอนสร้างออเดอร์ใน workers/manage.order.js (เฉพาะเมื่อ shipping_method_id == 1; method 2 = ฟรี)

ปลายทางไทย — เรต fixed ตามจำนวนชิ้น

นับเฉพาะสินค้าที่ไม่ใช่ is_shiping_free:

จำนวนชิ้นค่าส่ง (THB)
1 – 480
5 – 7100
≥ 8150

หมายเหตุ: API calculateShippingCost (controllers/v1/sm/dhl.js) มีสูตรไทยแบบอิงน้ำหนัก (eco_bkk/eco_upc + ส่วนเกิน kg ละ 25 บาท) แต่ route TH ถูก comment ปิดไว้ ใน routes/v1/sm/dhl.js

ปลายทางต่างประเทศ — lookup จากตาราง

ใช้ SmDHLService.calculateByRegion(weight, region) หาแถวใน sm_rate_shippings ที่ weight_min ≤ weight ≤ weight_max และ region ตรง แล้วใช้ค่า eco_upc (ถ้าน้ำหนักเกินช่วงสูงสุด ใช้ eco_upc ของแถว weight_max)

GET /api/v1/sm/dhl/shipping-cost/global?weight=1200&country=Japan®ion=ASIA
// → { success, ship_cost, country, region, weight }

8Tracking & Auto Status Update (Cron)

ไฟล์ crons/update-dhl-completed.js รันโดย PM2 ตาม schedule 45 23 * * * (ทุกวัน 23:45) ตั้งใน ecosystem.config.js (process e29-shop:update.dhl.completed.cron) จบงานแล้ว process.exit(0)

ทำ 2 รอบ: orderFromE29Shop() (ตาราง purchase_orders, dhl_type='local') และ orderFromE29ShopAdmin() (ตาราง am_purchase_orders)

ขั้นตอน

  1. SELECT tracking ที่ status='active' และ order อยู่สถานะ prepare หรือ shipped
  2. ขอ access token จาก DHL (เช็ค code 100000)
  3. วนเป็น batch ครั้งละ 100 เลข → POST Tracking (เช็ค code 200)
  4. map event status → อัปเดตสถานะออเดอร์ผ่าน updateStatusDhl(...)
DHL Event Statusความหมายอัปเดตสถานะออเดอร์เป็น
77206รับพัสดุ / กำลังนำส่งshipped (เฉพาะถ้าตอนนี้เป็น prepare)
77093นำส่งสำเร็จcompleted

เป็นแบบ Pull (polling) ไม่ใช่ Webhook — สถานะอัปเดตวันละครั้งเท่านั้น ทีมที่ต้อง near-realtime ควรพิจารณาเพิ่ม DHL webhook / ลดรอบ cron

การดูสถานะแบบ on-demand สำหรับ global ทำผ่าน controllers/v1/purchase.js → trackingList() ซึ่งคืน events ของ local แบบ live ส่วน global คืนเฉพาะข้อมูลที่เก็บไว้ (dhl_type: 'global')

10Replication Checklist

ขั้นตอนสำหรับนำ pattern นี้ไปใช้โปรเจกต์ใหม่:

  1. ขอ DHL account/credential ทั้ง eCommerce (clientId, password, pickup/sold account) และ Express/MyDHL (basic auth user/pass + shipper account number)
  2. ตั้ง environment variables 10 ตัวตามข้อ 3 (เก็บใน vault/CI secret อย่า commit)
  3. เพิ่ม config block dhl และ dhlGlobal ใน config loader
  4. สร้างตาราง shipping track (local/global), rate table, country table (อย่างน้อยต้องมี cca2, region, plt_allowed_in, no_postal_code) + seed ข้อมูลประเทศและเรต
  5. เปลี่ยนข้อมูล shipper ที่ hardcode ในตัวสร้าง global label (ที่อยู่บริษัท, VAT, account number, ขนาดกล่อง, description)
  6. ทำ endpoint สร้าง label ที่ branch ตาม cca2 == 'TH' → Local / มิฉะนั้น Global (อ้างอิง createTracking)
  7. ทำฟังก์ชันเก็บไฟล์ base64 → label/invoice + insert tracking row
  8. ตั้ง cron polling tracking + map event code (77206/77093) เป็นสถานะออเดอร์
  9. ทดสอบบน sandbox ของ DHL ก่อน แล้วค่อยสลับ endpoint เป็น production

11Caveats & Known Limitations

1. Secrets ใน .env ถูก commit ลง git — โปรเจกต์ต้นทางมี credential DHL จริงใน repo ควร rotate ค่าเหล่านั้นและถอดออกจาก git history; ทีมใหม่อย่าทำตาม pattern นี้

2. DHL Global ยังชี้ sandbox — DHL_GLOBAL_API_LABEL = .../mydhlapi/test/shipments ต้องเปลี่ยนเป็น production endpoint ก่อนใช้งานจริง

3. ข้อมูล shipper hardcode ในโค้ด (ที่อยู่, VAT, account 568963397, ขนาดกล่อง 35×38×1)

4. อัปเดตสถานะแบบ cron วันละครั้ง (23:45) ไม่ใช่ webhook — ดีเลย์ได้ถึง ~24 ชม.

5. Admin shop order ไม่รองรับส่งต่างประเทศ — controllers/v1/admin/shop/order.js โยน error “ระบบไม่รองรับการสร้าง label ส่งต่างประเทศ”

6. Error handling จำกัด — global label ไม่เช็ค business code (อาศัย axios throw เมื่อ HTTP error); cron จับ error รายตัวแล้ว log ต่อ ไม่มี retry/alert

7. axios เวอร์ชันเก่า (0.27.2) — ควรพิจารณาอัปเกรดเมื่อนำไปใช้ใหม่
